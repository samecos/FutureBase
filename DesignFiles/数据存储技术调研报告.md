## 1. 执行摘要

### 1.1 调研目标

本报告针对半自动化建筑设计平台的数据存储需求进行全面技术调研，涵盖以下核心领域：

- 几何数据（混凝土体块、墙体、设备等）的高效存储与查询
- 属性数据（外墙标识、防水类型、降板高度等）的管理
- 历史版本和撤销重做功能的实现
- 多专业并发操作的支持
- 用户账号隔离与数据安全

### 1.2 核心结论

| 技术领域 | 推荐方案 | 理由 |
|---------|---------|------|
| 几何数据存储 | **PostgreSQL + PostGIS** | 功能最全面，支持3D几何、IFC格式、R-tree索引 |
| 分布式数据库 | **YugabyteDB** 或 **CockroachDB** | 强一致性、PostgreSQL兼容、水平扩展 |
| 版本控制 | **Event Sourcing + 快照** | 完整审计、时间旅行、撤销重做支持 |
| 多租户隔离 | **Schema级隔离 + 行级RLS** | 平衡安全性与成本 |
| 缓存策略 | **Redis多级缓存** | 高性能、支持几何数据缓存 |
| 一致性策略 | **强一致性为主 + 最终一致性为辅** | 设计数据强一致，分析数据可接受延迟 |

## 2. 几何数据存储方案调研

### 2.1 空间数据库对比

#### 2.1.1 PostGIS (PostgreSQL扩展)

**技术特点**：
- 业界最成熟的空间数据库扩展，始于2005年
- 支持2D/3D/4D几何数据类型
- 提供3000+空间函数和操作符
- 严格遵循OpenGIS标准规范
- 支持R-tree、GiST、SP-GiST等多种空间索引

**性能表现**：
- 复杂空间分析：业界领先
- 大半径查询（>1000km）：优于MongoDB约3倍
- 几何关系计算：精度可控，支持平面与球面计算

**适用场景**：
- 需要复杂空间分析的建筑设计平台
- 3D几何数据管理
- 需要与其他GIS系统集成的场景

**参考数据**：
```sql
-- PostGIS 3D几何示例
CREATE TABLE building_elements (
    id SERIAL PRIMARY KEY,
    element_type VARCHAR(50),
    geometry GEOMETRY(GEOMETRYZ, 4326),  -- 3D几何
    properties JSONB
);

-- 创建3D空间索引
CREATE INDEX idx_building_geom_3d ON building_elements 
USING GIST(geometry gist_geometry_ops_3d);
```

#### 2.1.2 MongoDB Geospatial

**技术特点**：
- 原生支持GeoJSON格式
- 支持2dsphere和2d索引
- 提供基础地理查询功能（$near, $geoWithin等）
- 水平扩展能力强

**性能表现**：
- 点查询和交集查询：比PostGIS快3-6倍
- 简单查询延迟更低
- 功能有限（仅4种核心地理函数）

**适用场景**：
- 简单地理查询场景
- 需要高写入吞吐量的应用
- 快速原型开发

**参考数据**：
```javascript
// MongoDB 地理数据示例
db.building_elements.createIndex({ geometry: "2dsphere" });

db.building_elements.insertOne({
    element_type: "concrete_block",
    geometry: {
        type: "Polygon",
        coordinates: [[[...]]]  // GeoJSON格式
    },
    properties: {
        material: "C30",
        volume: 125.5
    }
});
```

#### 2.1.3 专用几何数据库对比

| 特性 | PostGIS | MongoDB Geo | SpatiaLite | GanosBase(阿里云) |
|------|---------|-------------|------------|-------------------|
| 3D几何支持 | 优秀 | 基础 | 良好 | 优秀 |
| IFC格式支持 | 通过扩展 | 需转换 | 需转换 | 原生支持 |
| 空间函数数量 | 3000+ | 4种 | 500+ | 2000+ |

### 2.3 几何索引策略

#### 2.3.1 R-tree索引

**原理**：
- 平衡树结构，用于空间数据的范围查询
- 将空间对象用最小边界矩形(MBR)表示
- 支持高效的范围查询和最近邻查询

**适用场景**：
- 2D/3D空间范围查询
- 碰撞检测
- 空间关系判断（相交、包含等）

**PostGIS实现**：
```sql
-- 创建R-tree索引
CREATE INDEX idx_elements_rtree ON building_elements 
USING GIST(geometry);

-- 3D R-tree索引
CREATE INDEX idx_elements_3d ON building_elements 
USING GIST(geometry gist_geometry_ops_nd);
```

#### 2.3.2 Quadtree/Octree索引

**原理**：
- 递归将空间划分为四等分（2D）或八等分（3D）
- 适用于点数据和均匀分布的场景
- 常用于LOD（细节层次）管理

**适用场景**：
- 大规模点云数据
- LOD层级管理
- 空间分块加载

#### 2.3.3 索引策略建议

| 数据类型 | 推荐索引 | 理由 |
|----------|----------|------|
| 建筑构件(3D) | R-tree (3D) | 支持复杂3D查询 |
| 设备点位 | Quadtree | 点数据优化 |
| 楼层平面 | R-tree (2D) | 标准空间查询 |
| 点云数据 | Octree | 海量点数据分块 |

### 2.4 大规模几何数据存储建议

#### 2.4.1 分区策略

```sql
-- 按项目分区
CREATE TABLE building_elements (
    id BIGSERIAL,
    project_id UUID,
    element_type VARCHAR(50),
    geometry GEOMETRY(GEOMETRYZ),
    created_at TIMESTAMP

-- 创建分区表

    FOR VALUES WITH (MODULUS 16, REMAINDER 0);

```

#### 2.4.2 存储优化

1. **几何简化**：存储多精度版本（LOD0-LOD3）
2. **压缩存储**：使用WKB二进制格式
3. **冷热分离**：历史版本归档到对象存储
4. **CDN加速**：glTF模型文件分发

## 3. 分布式数据库选型

### 3.1 NewSQL数据库对比

#### 3.1.1 CockroachDB

**技术特点**：
- 完全兼容PostgreSQL协议
- 默认Serializable隔离级别
- 基于Raft共识算法
- Range分区 + Multi-Raft架构

**性能数据**（基于TPC-C基准测试）：
- OLTP混合负载：~45K TPS
- 读密集型：~85K QPS
- 写密集型：~35K TPS
- P99延迟：50-100ms（地理分布式写入）

**优势**：
- 强一致性保证
- 自动故障恢复
- 云原生设计

**劣势**：
- 不支持线性一致性（因果一致性）
- 复杂查询性能一般

#### 3.1.2 TiDB

**技术特点**：
- MySQL协议兼容
- HTAP（混合事务/分析处理）
- 支持线性一致性
- TiKV + TiFlash双引擎

**性能数据**：
- OLTP混合负载：~52K TPS
- 读密集型：~95K QPS
- 写密集型：~40K TPS
- P99延迟：30-80ms

**优势**：
- 优秀的分析性能
- 线性一致性支持
- 成熟的生态（mixi2等大规模应用）

**劣势**：
- MySQL生态（非PostgreSQL）
- 需要更多运维经验

#### 3.1.3 YugabyteDB

**技术特点**：
- PostgreSQL兼容（YQL层）
- 双架构：YQL + DocDB
- 支持SQL和CQL两种API
- 云原生设计

**性能数据**：
- OLTP混合负载：~48K TPS
- 读密集型：~90K QPS
- 写密集型：~38K TPS
- P99延迟：40-90ms

**优势**：
- 最佳PostgreSQL兼容性
- 支持文档数据库接口
- 活跃的社区

**劣势**：
- 相对较新的项目
- 部分高级功能仍在开发

## 4. 版本控制和历史数据管理

### 4.1 时间旅行数据库

#### 4.1.1 PostgreSQL Temporal Tables

**系统版本表**：
```sql
-- 创建系统版本表（PostgreSQL 10+）
CREATE TABLE design_elements (
    id UUID PRIMARY KEY,
    element_type VARCHAR(50),
    geometry GEOMETRY,
    properties JSONB,
    valid_from TIMESTAMP GENERATED ALWAYS AS ROW START,
    valid_to TIMESTAMP GENERATED ALWAYS AS ROW END,
    PERIOD FOR SYSTEM_TIME (valid_from, valid_to)
) WITH (SYSTEM_VERSIONING = ON);

-- 查询历史版本
SELECT * FROM design_elements 
FOR SYSTEM_TIME AS OF '2024-01-15 10:00:00';
```

**局限性**：
- 仅支持行级版本
- 不支持复杂撤销重做
- 存储开销较大

### 4.4 撤销重做实现方案

#### 4.4.1 基于Event Sourcing的撤销重做

```python
# 撤销重做管理器
class UndoRedoManager:
    def __init__(self, event_store):
        self.event_store = event_store
        self.undo_stack = []
        self.redo_stack = []
    
    def execute(self, command):
        """执行命令并记录事件"""
        events = command.execute()
        self.event_store.append(events)
        self.undo_stack.append(events)
        self.redo_stack.clear()
    
    def undo(self):
        """撤销最后一个操作"""
        if not self.undo_stack:
            return None
        events = self.undo_stack.pop()
        compensations = self.create_compensations(events)
        self.event_store.append(compensations)
        self.redo_stack.append(events)
        return compensations
    
    def redo(self):
        """重做最后一个撤销的操作"""
        if not self.redo_stack:
            return None
        events = self.redo_stack.pop()
        self.event_store.append(events)
        self.undo_stack.append(events)
        return events
```

## 5. 多租户数据隔离方案

### 5.1 隔离方案对比

#### 5.1.1 数据库级别隔离（Database-per-Tenant）

```
┌─────────────────────────────────────────────────────────────┐
│                 数据库级别隔离架构                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────┐                                            │
│  │   应用层     │                                            │
│  └──────┬──────┘                                            │
│         │ 路由                                               │
│  ┌──────┴──────┐                                            │
│  │  动态数据源   │                                            │
│  └──────┬──────┘                                            │
│         │                                                    │
│    ┌────┼────┬────────┐                                     │
│    ↓    ↓    ↓        ↓                                     │
│  ┌───┐┌───┐┌───┐   ┌───┐                                   │
│  │DB1││DB2││DB3│...│DBn│  独立数据库实例                    │
│  └───┘└───┘└───┘   └───┘                                   │
│  租户A 租户B 租户C   租户N                                   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**优点**：
- 最高级别的数据隔离安全性
- 独立性能调优和备份策略
- 故障隔离，单租户问题不影响其他租户

**缺点**：
- 成本高（每个租户独立资源）
- 管理复杂度高
- 扩展性受限

**适用场景**：
- 大型企业客户
- 强合规要求行业（金融、医疗）
- 高安全需求客户（政府）

#### 5.1.2 Schema级别隔离

```sql
-- PostgreSQL Schema隔离
CREATE SCHEMA tenant_acme_corp;
CREATE SCHEMA tenant_tech_ltd;

-- 为每个租户创建独立表
CREATE TABLE tenant_acme_corp.building_elements (...);
CREATE TABLE tenant_tech_ltd.building_elements (...);

-- 使用RLS增强安全性
ALTER TABLE tenant_acme_corp.building_elements 
ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation_policy
ON tenant_acme_corp.building_elements
USING (current_setting('app.tenant_id') = 'acme_corp');
```

**优点**：
- 较好的数据隔离
- 资源利用率较高
- 管理相对简单

**缺点**：
- Schema变更需要同步所有租户
- 单实例故障影响所有租户

**适用场景**：
- 中等规模租户数量
- 需要平衡隔离性和资源利用率

#### 5.1.3 行级别隔离（Tenant ID字段）


**适用场景**：
- 大量小租户
- 成本敏感型SaaS
- 标准化需求高

#### 6.3.2 缓存失效机制

```python
# 缓存失效管理器
class CacheInvalidator:
    def __init__(self, redis_client):
        self.redis = redis_client
    
    def invalidate_element(self, project_id, element_id):
        """失效单个构件缓存"""
        keys = [
            f"project:{project_id}:element:{element_id}:geometry",
            f"project:{project_id}:element:{element_id}:props",
            f"project:{project_id}:spatial:index"
        ]
        self.redis.delete(*keys)
        # 发布失效消息
        self.redis.publish('cache:invalidate', {
            'project_id': project_id,
            'element_id': element_id
        })
    
    def invalidate_project(self, project_id):
        """失效整个项目缓存"""
        pattern = f"project:{project_id}:*"
        for key in self.redis.scan_iter(pattern):
            self.redis.delete(key)
```

### 7.3 冲突检测和解决

#### 7.3.1 版本向量

```python
class VersionVector:
    """版本向量用于检测并发冲突"""
    
    def __init__(self):
        self.versions = {}  # {replica_id: version}
    
    def increment(self, replica_id):
        self.versions[replica_id] = self.versions.get(replica_id, 0) + 1
    
    def compare(self, other):
        """
        返回值:
        -1: self < other (self是祖先)
         0: self == other (相同版本)
         1: self > other (other是祖先)
         2: 并发冲突
        """
        dominates = False
        dominated = False
        
        all_replicas = set(self.versions.keys()) | set(other.versions.keys())
        
        for replica in all_replicas:
            v1 = self.versions.get(replica, 0)
            v2 = other.versions.get(replica, 0)
            
            if v1 > v2:
                dominates = True
            elif v2 > v1:
                dominated = True
        
        if dominates and dominated:
            return 2  # 冲突
        elif dominates:
            return 1
        elif dominated:
            return -1
        else:
            return 0
```

#### 7.3.2 冲突解决策略

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| Last-Write-Wins | 时间戳最新的获胜 | 简单场景 |
| 合并冲突 | 自动合并可合并的变更 | 结构化数据 |
| 用户决策 | 提示用户选择 | 重要数据 |
| 操作转换 | OT算法合并操作 | 协作编辑 |

#### 9.2.2 项目表

```sql
-- 项目表
CREATE TABLE projects (

    name VARCHAR(200) NOT NULL,
    description TEXT,
    status VARCHAR(20) DEFAULT 'draft',
    created_by UUID NOT NULL,
```

#### 9.2.3 构件表（PostGIS）

```sql
-- 构件表
CREATE TABLE elements (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id UUID NOT NULL REFERENCES projects(id),
    element_type VARCHAR(50) NOT NULL,
    name VARCHAR(200),
    
    -- 3D几何数据 (WKB格式)
    geometry GEOMETRY(GEOMETRYZ, 4326),
    
    -- 包围盒（用于快速筛选）
    bbox GEOMETRY(POLYGON, 4326),
    
    -- 属性数据（JSONB）
    properties JSONB DEFAULT '{}',
    
    -- 版本控制
    version INTEGER DEFAULT 1,
    is_deleted BOOLEAN DEFAULT FALSE,
    
    -- 审计字段
    created_by UUID NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_by UUID,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 空间索引
CREATE INDEX idx_elements_geom ON elements USING GIST(geometry);
CREATE INDEX idx_elements_bbox ON elements USING GIST(bbox);
CREATE INDEX idx_elements_3d ON elements USING GIST(geometry gist_geometry_ops_nd);

-- 属性索引
CREATE INDEX idx_elements_type ON elements(element_type);
CREATE INDEX idx_elements_project ON elements(project_id);
CREATE INDEX idx_elements_properties ON elements USING GIN(properties);

-- 行级安全策略
ALTER TABLE elements ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation_policy ON elements
USING (project_id IN (
    SELECT id FROM projects WHERE tenant_id = current_setting('app.tenant_id')::UUID
));
```

#### 9.2.4 版本历史表

```sql
-- 构件版本历史表
CREATE TABLE element_versions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    element_id UUID NOT NULL REFERENCES elements(id),
    version INTEGER NOT NULL,
    
    -- 变更类型: create, modify, delete
    change_type VARCHAR(20) NOT NULL,
    
    -- 变更内容
    geometry GEOMETRY(GEOMETRYZ, 4326),
    properties JSONB,
    property_changes JSONB,  -- 仅存储变更的字段
    
    -- 审计
    changed_by UUID NOT NULL,
    changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    change_reason TEXT,
    
    -- 唯一约束
    UNIQUE(element_id, version)
);

-- 索引
CREATE INDEX idx_versions_element ON element_versions(element_id);
CREATE INDEX idx_versions_time ON element_versions(changed_at);
```

#### 9.2.5 事件存储表

```sql
-- 事件存储表（Event Sourcing）
CREATE TABLE events (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    
    -- 事件元数据
    event_type VARCHAR(100) NOT NULL,
    aggregate_type VARCHAR(50) NOT NULL,  -- 'element', 'project', etc.
    aggregate_id UUID NOT NULL,
    version INTEGER NOT NULL,
    
    -- 事件数据
    payload JSONB NOT NULL,
    metadata JSONB DEFAULT '{}',
    
    -- 审计
    created_by UUID NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    -- 全局排序（用于事件重放）
    global_sequence BIGSERIAL
);

-- 索引
CREATE INDEX idx_events_aggregate ON events(aggregate_type, aggregate_id, version);
CREATE INDEX idx_events_global_seq ON events(global_sequence);
CREATE INDEX idx_events_type ON events(event_type);
CREATE INDEX idx_events_time ON events(created_at);

-- 唯一约束：确保版本顺序
CREATE UNIQUE INDEX idx_events_version ON events(aggregate_type, aggregate_id, version);
```

#### 9.2.6 快照表

```sql
-- 快照表（用于加速状态重建）
CREATE TABLE snapshots (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    aggregate_type VARCHAR(50) NOT NULL,
    aggregate_id UUID NOT NULL,
    version INTEGER NOT NULL,
    
    -- 序列化状态
    state_data JSONB NOT NULL,
    
    -- 元数据
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    event_count INTEGER,  -- 从此快照开始的事件数量
    
    UNIQUE(aggregate_type, aggregate_id, version)
);

-- 索引
CREATE INDEX idx_snapshots_aggregate ON snapshots(aggregate_type, aggregate_id);
```

### 10.1 技术风险

| 风险 | 影响 | 可能性 | 缓解措施 |
|------|------|--------|----------|
| PostGIS性能瓶颈 | 高 | 中 | 分区、索引优化、读写分离 |
| 分布式事务复杂性 | 中 | 高 | 采用Saga模式、充分测试 |
| 数据迁移困难 | 高 | 中 | 设计可逆迁移、备份策略 |
| 缓存一致性问题 | 中 | 中 | 多级缓存策略、主动失效 |
| 多租户安全漏洞 | 高 | 低 | RLS策略、安全审计 |
