## 1. 执行摘要

### 1.1 调研目标

本报告针对半自动化建筑设计平台的脚本执行引擎进行技术调研，旨在为平台选择合适的脚本语言、执行引擎、任务调度系统和安全隔离方案。

### 1.2 核心需求分析

| 需求维度 | 具体要求 |
|---------|---------|
| 脚本编写 | 工程师和程序员编写自动化深化脚本 |
| 脚本调度 | 后端调度执行，支持依赖管理 |
| 属性依赖 | 脚本运行依赖前期属性定义 |
| 结果反馈 | 执行结果反馈到设计模型 |
| 安全隔离 | 多租户环境下安全执行用户代码 |
| 性能要求 | 支持大规模并行计算 |

### 1.3 推荐技术栈（概要）

| 组件 | 推荐方案 | 备选方案 |
|-----|---------|---------|
| 脚本语言 | Python | JavaScript/TypeScript |
| 执行引擎 | Docker + gVisor | WebAssembly |
| 任务调度 | Temporal | Celery + 自定义调度器 |
| 代码编辑器 | CodeMirror 6 | Monaco Editor |
| 安全隔离 | gVisor + seccomp + cgroups | Firecracker MicroVM |

## 2. 脚本语言选型

### 2.1 Python（推荐）

#### 优势

| 维度 | 说明 |
|-----|------|
| 行业生态 | 建筑设计行业主流语言，Rhino/Grasshopper、Blender、FreeCAD等均支持Python |
| 库生态 | 丰富的科学计算库（NumPy、SciPy）、几何处理库（Shapely、Trimesh） |
| 学习曲线 | 语法简洁，设计师群体接受度高 |
| 嵌入式支持 | CPython C API、PyPy沙箱等成熟方案 |
| 调试工具 | pdb、ipdb、VS Code等完善调试生态 |

#### 劣势

| 维度 | 说明 |
|-----|------|
| 性能 | GIL限制，纯CPU密集型任务需借助C扩展 |
| 沙箱安全 | CPython沙箱实现复杂，需配合OS级隔离 |
| 启动时间 | 解释器启动相对较慢 |

#### 适用场景
- 几何算法实现
- 参数化设计脚本
- 与现有CAD工具集成

### 2.2 JavaScript/TypeScript

#### 优势

| 维度 | 说明 |
|-----|------|
| Web原生 | 与前端技术栈无缝集成 |
| V8引擎 | 高性能JIT编译，执行速度快 |
| 沙箱安全 | V8隔离模型成熟，内存安全有保障 |
| 类型系统 | TypeScript提供静态类型检查 |

#### 劣势

| 维度 | 说明 |
|-----|------|
| 行业生态 | 建筑设计领域库生态不如Python丰富 |
| 数值计算 | 科学计算库不如Python成熟 |
| 几何处理 | 缺乏工业级几何内核库 |

#### 适用场景
- 前端交互脚本
- 轻量级数据处理
- 与WebGL可视化集成

### 2.3 Lua

#### 优势

| 维度 | 说明 |
|-----|------|
| 轻量级 | 解释器体积小（~200KB），启动快 |
| 嵌入友好 | C API简洁，易于嵌入宿主程序 |
| 游戏行业 | 游戏脚本标准，有成熟沙箱实践 |

#### 劣势

| 维度 | 说明 |
|-----|------|
| 生态局限 | 科学计算、几何处理库稀缺 |
| 人才储备 | 建筑设计领域Lua人才较少 |

#### 适用场景
- 配置型脚本
- 性能敏感的简单计算

### 2.4 领域特定语言（DSL）

#### 可行性分析

| 维度 | 评估 |
|-----|------|
| 开发成本 | 高，需设计语法、实现解释器/编译器 |
| 维护成本 | 高，需持续演进语言特性 |
| 用户学习 | 中，领域专家可快速上手 |
| 灵活性 | 低，难以处理边界场景 |

#### 建议
- **短期**：基于Python/JS的嵌入式DSL（装饰器、链式API）
- **长期**：如业务稳定后可考虑专用DSL

### 2.5 语言选型结论

```
推荐优先级：
1. Python（主选）- 行业生态优势显著
2. JavaScript/TypeScript（备选）- Web集成优势
3. Lua（特定场景）- 轻量级需求
4. 自研DSL（长期规划）- 业务成熟后评估
```

```

┌─────────────────────────────────────┐

│  │   Model/Data Access Layer   │   │
│  └─────────────────────────────┘   │
```

| 特性 | 评估 |
|-----|------|

#### PyPy Sandbox

| 特性 | 评估 |
|-----|------|

| 安全机制 | 所有IO通过父进程代理 |
| 性能 | 接近原生，JIT优化 |
| 成熟度 | 实验性，生产环境需谨慎 |

```
│  │  │  │  │    Python Runtime       │   │   │   │   │
│  │  │  │  │  ┌─────────────────┐   │   │   │   │   │
```

| 特性 | 评估 |
|-----|------|

| 启动时间 | ~100ms |

#### Firecracker MicroVM

| 特性 | 评估 |
|-----|------|
| 隔离级别 | 硬件级，专用内核 |
| 启动时间 | ~125ms |
| 内存开销 | <5MB |
| 适用场景 | 高安全要求的多租户环境 |

## 4. 任务调度系统

### 4.1 工作流引擎对比

#### Temporal（推荐）

```
核心架构：
┌─────────────────────────────────────────────────────────┐
│                  Temporal Service                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐    │
│  │  Frontend   │  │   History   │  │  Matching   │    │
│  │   Gateway   │  │   Service   │  │  Service    │    │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘    │
│         └─────────────────┼─────────────────┘          │
│                           ↓                           │
│                  ┌─────────────────┐                  │
│                  │  Persistence    │                  │
│                  │  (PostgreSQL/   │                  │
│                  │   Cassandra)    │                  │
│                  └─────────────────┘                  │
└─────────────────────────────────────────────────────────┘
                            ↑
┌─────────────────────────────────────────────────────────┐
│                    Worker Pool                          │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐   │
│  │ Worker 1 │ │ Worker 2 │ │ Worker 3 │ │ Worker N │   │
│  │ ┌──────┐ │ │ ┌──────┐ │ │ ┌──────┐ │ │ ┌──────┐ │   │
│  │ │Task A│ │ │ │Task B│ │ │ │Task C│ │ │ │Task D│ │   │
│  │ └──────┘ │ │ └──────┘ │ │ └──────┘ │ │ └──────┘ │   │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘   │
└─────────────────────────────────────────────────────────┘
```

| 特性 | Temporal | Apache Airflow |
|-----|----------|----------------|
| 核心模型 | Durable Execution | DAG-based Scheduling |
| 状态持久化 | 自动（Event Sourcing） | 外部存储（XCom） |
| 故障恢复 | Replay from history | Retry from scratch |
| 长等待支持 | Durable timers（天/月） | Sensors占用worker |
| 执行语义 | Exactly-once | At-least-once |
| 多语言SDK | Go/Java/Python/TS/.NET/PHP/Ruby | Python only |
| 适用场景 | 事务性业务工作流 | 批处理数据管道 |

**关键优势：**
- 代码优先：用普通代码定义工作流
- 自动状态恢复：worker崩溃后自动恢复
-  Saga模式：支持分布式事务补偿

#### Apache Airflow

| 特性 | 评估 |
|-----|------|
| 成熟度 | 高，2014年发布，大规模验证 |
| 生态 | 60+预置provider（AWS/GCP/Snowflake） |
| 可视化 | 丰富的DAG可视化界面 |
| 限制 | 批处理为主，长运行工作流支持弱 |

## 5. 安全隔离方案

### 5.1 隔离技术层次

```
隔离层次金字塔：

        ┌─────────────┐
        │   硬件隔离   │  ← Firecracker/Kata Containers
        │  (MicroVM)  │
        ├─────────────┤
        │  用户空间内核 │  ← gVisor
        │  (runsc)    │
        ├─────────────┤
        │  系统调用过滤 │  ← seccomp-BPF
        │  (syscall)  │
        ├─────────────┤
        │  命名空间隔离 │  ← namespaces
        │  (namespace)│
        ├─────────────┤
        │  资源限制    │  ← cgroups
        │  (cgroups)  │
        ├─────────────┤
        │  权限控制    │  ← capabilities
        │  (cap)      │
        └─────────────┘
```

### 5.2 沙箱技术详解

#### gVisor架构

```
gVisor安全边界：
┌─────────────────────────────────────────────────────────┐
│  Untrusted Code                                         │
│       ↓                                                 │
│  ┌─────────────────────────────────────────────────┐   │
│  │              Sentry (User-space Kernel)          │   │
│  │  • 实现Linux系统调用子集                          │   │
│  │  • Go语言实现，内存安全                            │   │
│  │  • 拦截所有syscall                                │   │
│  └─────────────────────────────────────────────────┘   │
│       ↓                                                 │
│  ┌─────────────────────────────────────────────────┐   │
│  │              Gofer (File Proxy)                  │   │
│  │  • 文件系统访问代理                               │   │
│  │  • 9P协议通信                                     │   │
│  └─────────────────────────────────────────────────┘   │
│       ↓                                                 │
│  ┌─────────────────────────────────────────────────┐   │
│  │           Host Kernel (Limited Syscalls)         │   │
│  │  • 仅允许~20个syscall                            │   │
│  │  • 极小的攻击面                                   │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

#### seccomp-BPF

| 特性 | 说明 |
|-----|------|
| 默认策略 | Docker默认阻止~40个危险syscall |
| 自定义 | 可定义允许/拒绝的syscall白名单 |
| 限制 | 同一内核，允许的syscall仍有攻击面 |

#### 资源限制（cgroups）

```yaml
# 资源限制配置示例
resources:
  limits:
    cpu: "2"
    memory: "4Gi"
    ephemeral-storage: "10Gi"
  requests:
    cpu: "100m"
    memory: "512Mi"
  
# 时间限制
timeout:
  hard_limit: 300s
  soft_limit: 60s

# 进程限制
pids:
  max: 100
```

### 5.5 安全方案推荐

```
多层防御策略：
┌─────────────────────────────────────────────────────────┐
│  Layer 1: 代码审查                                       │
│  ├── 静态分析（Bandit/Semgrep）                          │
│  └── 危险函数黑名单                                      │
```

### 7.4 结果缓存

| 缓存级别 | 策略 | TTL |
|---------|------|-----|
| L1 - 内存 | 热点结果 | 5分钟 |
| L2 - Redis | 计算结果 | 1小时 |
| L3 - 对象存储 | 大文件输出 | 7天 |

```python
# 缓存键设计
def generate_cache_key(script_id, model_hash, params):
    """
    缓存键 = 脚本ID + 模型哈希 + 参数哈希
    """
    content = f"{script_id}:{model_hash}:{json.dumps(params, sort_keys=True)}"
    return hashlib.sha256(content.encode()).hexdigest()
```
