## 1. 技术选型调研

### 1.1 微服务框架对比

| 框架 | 语言 | 性能 | 学习曲线 | 生态成熟度 | 适用场景 |
|------|------|------|----------|------------|----------|
| **Spring Cloud** | Java | 中等 | 中等 | ★★★★★ | 企业级大型系统 |
| **Go Micro** | Go | 高 | 中等 | ★★★☆☆ | 高性能微服务 |
| **Node.js (Express/Fastify)** | JavaScript | 中高 | 低 | ★★★★☆ | I/O密集型应用 |
| **Micronaut** | Java | 高 | 中等 | ★★★☆☆ | 云原生应用 |
| **Go kit** | Go | 极高 | 较高 | ★★★☆☆ | 低延迟系统 |

**详细分析**:

- **Spring Cloud**: 拥有最成熟的生态系统，超过60%的开发者选择。提供完整的微服务解决方案（服务发现、配置中心、熔断器等），但内存占用较高（典型服务>200MB），启动时间较长。

- **Go Micro/Go kit**: Go语言的goroutines可高效管理数千并发操作，内存占用极低（<50MB），部署包<20MB，CPU使用量比JVM方案低30%。特别适合延迟敏感的流式系统。

- **Node.js**: 事件驱动、非阻塞I/O模型，可服务2-3倍于线程模型的并发用户。FastAPI等框架在并发环境下比传统Flask快3-5倍。

**推荐**: 对于建筑设计平台，建议采用**多语言微服务架构**:
- 核心协作服务: Go (高性能实时通信)
- 业务逻辑服务: Java/Spring Boot (成熟稳定)
- 脚本执行服务: Go/Rust (安全隔离)

### 1.2 实时协作技术方案

#### OT vs CRDT 对比

| 特性 | OT (Operational Transformation) | CRDT (Conflict-free Replicated Data Types) |
|------|----------------------------------|-------------------------------------------|
| **核心思想** | 操作转换，保持操作顺序 | 无冲突复制数据结构 |
| **中心化** | 需要中央服务器协调 | 可完全去中心化 |
| **冲突解决** | 服务器端转换操作 | 数学保证无冲突 |
| **离线支持** | 有限 | 优秀 |
| **实现复杂度** | 高 | 中等 |
| **扩展性** | 受中央服务器限制 | 良好 |
| **典型应用** | Google Docs, 早期协作系统 | Figma, Apple Notes |

**技术选型建议**:

1. **CRDT为主**: 现代协作系统首选，Figma的成功验证了CRDT在图形协作场景的有效性
2. **混合模式**: 复杂操作使用OT保证顺序，简单属性使用CRDT
3. **参考实现**: Yjs, Automerge (JavaScript); diamond-types (Rust)

#### WebSocket vs SSE 对比

| 特性 | WebSocket | Server-Sent Events (SSE) |
|------|-----------|-------------------------|
| **通信模式** | 全双工 | 服务器单向推送 |
| **协议** | TCP + 自定义帧 | HTTP/1.1 或 HTTP/2 |
| **延迟** | <10ms | 5-20ms |
| **穿透性** | 需处理代理/防火墙 | 基于HTTP，穿透性好 |
| **重连机制** | 需自行实现 | 浏览器原生支持 |
| **二进制支持** | 原生支持 | Base64编码 |
| **适用场景** | 实时双向通信 | 服务器推送更新 |

**推荐**: 
- **WebSocket**: 用于实时协作编辑、光标同步、即时消息
- **SSE**: 用于通知推送、进度更新、单向数据流

### 1.3 消息队列选型

| 特性 | Apache Kafka | RabbitMQ | NATS JetStream |
|------|--------------|----------|----------------|
| **吞吐量** | 500K-1M msg/s | 50K-100K msg/s | 200K-400K msg/s |
| **延迟(P99)** | 10-50ms | 5-20ms | 0.5-5ms |
| **持久化** | 始终持久化 | 可配置 | 可选 |
| **交付保证** | At-least-once / Exactly-once | At-least-once | Exactly-once |
| **内存占用** | 8-16GB | 100-500MB | 10-50MB |
| **协议** | 自定义二进制 | AMQP/MQTT/STOMP | 自定义二进制 |
| **流处理** | 原生支持 | 需插件 | 有限支持 |
| **集群模式** | 分区复制 | 镜像队列/仲裁队列 | RAFT共识 |

**选型建议**:

1. **Apache Kafka**: 事件溯源、审计日志、大数据分析场景
2. **RabbitMQ**: 任务队列、路由复杂的企业集成场景
3. **NATS JetStream**: 实时消息、轻量级事件总线、资源受限环境

**建筑设计平台推荐**:
- **主消息队列**: Apache Kafka (事件溯源、操作日志)
- **实时通知**: NATS (低延迟、轻量级)
- **任务队列**: RabbitMQ (脚本执行、渲染任务)

### 1.4 API设计风格

| 特性 | REST | GraphQL | gRPC |
|------|------|---------|------|
| **协议** | HTTP/1.1 | HTTP/1.1 | HTTP/2 |
| **数据格式** | JSON/XML | JSON | Protocol Buffers |
| **性能** | 中等 | 中高 | 极高 |
| **延迟** | 180ms (典型) | 95ms (典型) | 18ms (典型) |
| **带宽效率** | 基准 | 减少50% | 减少78% |
| **缓存支持** | 优秀 | 有限 | 良好 |
| **浏览器支持** | 完美 | 完美 | 需代理 |
| **学习曲线** | 低 | 中等 | 较高 |
| **适用场景** | 公共API | 复杂查询 | 微服务间通信 |

**性能基准数据** (来自2025年测试):
- gRPC比REST快**6.3倍**
- gRPC使用**46%更少**的CPU
- gRPC处理**7倍**更高吞吐量

**推荐架构**:
```
┌─────────────────────────────────────────────────────────────┐
│                        API 网关层                            │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐      │
│  │   REST API   │  │  GraphQL API │  │  gRPC Gateway│      │
│  │  (外部客户端) │  │  (BFF聚合)   │  │  (内部服务)   │      │
│  └──────────────┘  └──────────────┘  └──────────────┘      │
└─────────────────────────────────────────────────────────────┘
```

- **REST**: 第三方集成、移动端API
- **GraphQL**: 前端数据聚合、灵活查询
- **gRPC**: 微服务间内部通信

## 3. 关键技术挑战识别

### 3.1 几何数据的序列化和传输

**挑战**:
- 几何数据体积大（复杂模型可达GB级）
- 传输带宽限制
- 序列化/反序列化性能

**解决方案对比**:

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **Protocol Buffers** | 高效、跨语言 | 需预定义Schema | 结构化数据 |
| **FlatBuffers** | 零拷贝、快速访问 | 灵活性较低 | 大规模几何数据 |
| **glTF** | 3D标准、压缩好 | 特定于图形 | 3D模型传输 |
| **自定义二进制** | 最优性能 | 维护成本高 | 核心高频数据 |

**推荐策略**:
1. 核心几何数据: FlatBuffers (零拷贝优势)
2. 场景描述: glTF (行业标准)
3. 元数据: Protocol Buffers
4. 增量更新: 自定义二进制格式

### 3.2 大规模并发编辑的性能瓶颈

**潜在瓶颈**:
1. **操作广播**: N个用户同时编辑，操作冲突概率O(N²)
2. **状态同步**: 文档状态随操作增长
3. **渲染压力**: 客户端频繁重绘

**优化策略**:

| 策略 | 实现方式 | 预期效果 |
|------|----------|----------|
| **操作批处理** | 100ms窗口聚合操作 | 减少50%消息量 |
| **增量更新** | 仅传输变更区域 | 减少80%数据量 |
| **空间分区** | 八叉树划分场景 | 局部更新 |
| **LOD传输** | 根据视距简化几何 | 减少90%初始数据 |

### 3.3 脚本执行的安全隔离

**隔离方案对比**:

| 方案 | 隔离级别 | 启动时间 | 资源开销 | 适用场景 |
|------|----------|----------|----------|----------|
| **Docker容器** | 进程级 | 秒级 | 中等 | 通用隔离 |
| **WebAssembly** | 沙箱级 | 毫秒级 | 低 | 轻量级脚本 |
| **gVisor** | 系统调用拦截 | 秒级 | 中等 | 高安全需求 |
| **MicroVM** | 硬件虚拟化 | 100ms级 | 较高 | 强隔离 |
| **Seccomp** | 系统调用过滤 | 无 | 低 | 简单限制 |

**推荐架构**:
```
┌─────────────────────────────────────────────────────────────┐
│                    脚本执行服务                              │
│  ┌──────────────────────────────────────────────────────┐  │
│  │              调度器 (资源管理/队列)                    │  │
│  └──────────────────────────────────────────────────────┘  │
│                         ↓                                   │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐      │
│  │ WebAssembly  │  │   Docker     │  │   MicroVM    │      │
│  │  (轻量脚本)   │  │  (复杂任务)   │  │  (高危操作)   │      │
│  └──────────────┘  └──────────────┘  └──────────────┘      │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 跨服务数据一致性

**一致性模型选择**:

| 模型 | 保证 | 延迟 | 可用性 | 适用场景 |
|------|------|------|--------|----------|
| **强一致性** | 读最新写 | 高 | 低 | 金融交易、权限 |
| **最终一致性** | 最终收敛 | 低 | 高 | 用户状态、统计 |
| **因果一致性** | 因果顺序 | 中等 | 中等 | 协作编辑 |
| **读己所写** | 自己写可见 | 低 | 高 | 个人设置 |

**CAP定理应用**:
- **CP系统**: 权限验证、项目配置 (强一致性优先)
- **AP系统**: 用户在线状态、操作日志 (可用性优先)

## 5. 性能基准调研

### 5.1 协作系统响应时间要求

| 操作类型 | 目标延迟 | 可接受延迟 | 测量方式 |
|----------|----------|------------|----------|
| **本地操作反馈** | <16ms | <50ms | 端到端 |
| **操作广播** | <100ms | <300ms | 服务端处理 |
| **冲突解决** | <200ms | <500ms | 完整流程 |
| **文档加载** | <1s | <3s | 初始加载 |
| **历史回溯** | <500ms | <2s | 状态恢复 |

**参考标准**:
- Figma: 操作延迟<100ms
- Google Docs: 协作延迟<200ms
- 游戏行业: 输入响应<16ms (60fps)

### 5.2 几何数据处理吞吐量需求

**预估数据规模**:

| 场景 | 数据量 | 并发用户 | 吞吐量要求 |
|------|--------|----------|------------|
| **小型项目** | 10-100MB | 5-10 | 10 ops/s |
| **中型项目** | 100MB-1GB | 10-30 | 50 ops/s |
| **大型项目** | 1-10GB | 30-100 | 200 ops/s |
| **超大型项目** | >10GB | 100+ | 500+ ops/s |

**存储需求**:
- 每个项目平均: 500MB
- 版本历史(50个版本): 25GB
- 1000个项目: 25TB

### 5.3 脚本执行并发规模预估

| 场景 | 并发任务 | 执行时长 | 资源需求 |
|------|----------|----------|----------|
| **参数化设计** | 10-50 | 秒级 | 2-4 CPU |
| **批量导出** | 5-20 | 分钟级 | 4-8 CPU |
| **渲染任务** | 1-10 | 小时级 | 8-16 CPU + GPU |
| **仿真计算** | 1-5 | 小时级 | 16+ CPU |

**资源规划建议**:
- 脚本执行节点: 可水平扩展
- 渲染农场: GPU集群
- 任务队列: 优先级调度

| **Kafka** | 低 | 低 | 低 |
| **自研CRDT** | 高 | 高 | 高 |

1. 优先选择社区活跃的开源项目
